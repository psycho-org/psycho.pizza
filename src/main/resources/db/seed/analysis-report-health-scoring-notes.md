## 감점 판단 원칙

- 단순한 설정 변경 자체는 감점 이벤트가 아니다.
- `assignee 변경`, `due date 변경`, `sprint 기간 변경`, `project 변경`은 그 자체로는 중립적 운영 행위일 수 있다.
- 스프린트 시작일보다 늦게 착수한 사실만으로는 감점하지 않는다.
- 업무 간 선행관계, 순차 진행, 의도된 후속 착수는 정상 흐름으로 볼 수 있다.
- 실제 감점은 아래처럼 **영향이 발생했을 때** 부여한다.
  - 다음 일정 지연 발생
  - 전체 업무 진척 저하
  - 특정 인원에게 업무 과부하 집중
  - 스프린트 종료 시 미완료 또는 품질 저하로 귀결
- 즉, 로그는 원인 후보이고 감점은 결과/영향 기준으로 판단한다.

## 횟수형 이벤트 공통 기준

- 횟수가 중요한 이벤트는 동일한 구간 기준을 적용한다.
- 공통 구간:
  - `1건`: 단건 발생
  - `2~3건`: 반복 발생
  - `3건 초과`: 빈번 발생
- 기존 메모의 `1건 / 1~3건 / 3건 초과` 표현은 구간이 겹치므로, 문서에서는 겹치지 않게 `1건 / 2~3건 / 3건 초과`로 통일한다.
- 단, 이 횟수 기준도 단독 감점 트리거가 아니라 실제 영향과 함께 해석한다.

## 1. End-of-Sprint Event

> Sprint 종료 시 점수에 반영되는 이벤트

- 흐름 위반: 일이 앞으로 안 가는 경우
- 정체: 병목
- 계획 불안정: 예측 가능성 하락
- 품질/협업 마찰: 팀 합의 비용 증가

## 2. Ongoing Event

> 스프린트 중에 점수에 반영되는 이벤트

- 진행 중 발생하는 상태 변경, 일정 변경, 담당자 변경, 범위 변동 등의 누적 이벤트

## 가중치 메모 (1~5)

### 1. Sprint 종료 시 점수에 반영되는 이벤트

- Assignee가 지정되지 않음
  - 4
- CSG가 아닌 프로젝트/태스크 있음
  - 3
- CSG에서도 canceled와 done 비율이 2:8 이상인 경우
  - 4
- 스프린트에 프로젝트가 하나도 포함되지 않은 경우 && 프로젝트에 task가 하나도 없는 경우
  - 리포팅하지 않음
- 프로젝트에 task가 하나도 없는 경우
  - 3

### 2. 스프린트 중에 점수에 반영되는 이벤트

#### 기한 관련 이벤트

- Due date 지나서 done
  - 2~3
- Due date 반복적 연장으로 인해 실제 일정 지연 또는 마감 실패 발생
  - `1건 / 2~3건 / 3건 초과` 공통 기준 적용
- Sprint 진행 기간 동안 Sprint 기간이 변경되어 전체 계획 안정성이 하락한 경우
  - `1건 / 2~3건 / 3건 초과` 공통 기준 적용

#### 상태 관련 이벤트

- 리워크 (CSG에서 이전 상태로 돌아옴)
  - 3
- Task가 TODO에서 바로 DONE으로 가는 경우
  - 1~3
- 특정 상태로 오래 방치됨, 리뷰 병목
  - 4
- Canceled가 됨
  - 2~4
  - 추후 다시 수행 완료 시 감점 완화 가능
- DONE -> IN_PROGRESS / DONE -> TODO / IN_PROGRESS -> TODO 또는 BACKLOG
  - 별도 감점 대상
- 스프린트 중 추가/삭제된 Task 수
  - 5

#### 담당자 관련 이벤트

- Assignee 변경이 반복되어 인수인계 비용 증가, 지연, 병목이 발생한 경우
  - `1건 / 2~3건 / 3건 초과` 공통 기준 적용
- 특정인에게 업무 집중도 높음
  - 3
- 인원 수 대비 Work In Progress가 너무 많은 경우
  - 3
  - `Penalty = (InProgress / ActiveMembers) × (InProgress / TotalTasks)`
- Task의 프로젝트 변경으로 진척 저하 또는 재작업이 발생한 경우
  - `1건 / 2~3건 / 3건 초과` 공통 기준 적용

### 추가 메모

- 상황에 따라 이슈 신규 발행 / 이슈 수정의 의미가 달라질 수 있음
- 이슈 커밋이 일어나지 않는 상황도 추후 고려 가능
- 데일리 스크럼 단위 트래킹 필요성 존재
- 세레모니 개념은 개발자 전용 애자일 개념으로 과도하게 좁히지 않도록 축소해서 해석

## 추가 감점 이벤트 후보

### A. 일정 / 예측 정확도 관련

1. 커밋하지 않은 Task가 Done의 40% 이상 차지
   - 3
2. 마지막 2일 내 Done 몰림 (전체 Done의 50% 이상)
   - 3
3. Due date 미설정 Task가 30% 이상
   - 2
4. 스프린트 중 목표 변경 (Goal 수정)
   - 수정할 때마다 3

### B. 태스크 품질 관련

1. Description 없는 Task 비율 30% 이상
   - 2

### D. 프로세스 안정성 관련

1. WIP Limit 초과 상태가 N일 이상 지속
   - 하루당 3
   - 기준은 `인원 수 * 1.2~1.5`, 추후 커스텀 가능하게 확장
2. 스프린트 중 삭제된 Task 비율 30% 이상
   - 4

## 카테고리

1. Completion (완결력)
2. Flow (흐름)
3. Stability (계획 안정성)
4. Ownership (책임/리소스)

## Sprint Health Scoring 최종 사용 지표 표

### 1. Sprint 종료 시 반영 이벤트

| 카테고리 | 이벤트 | 조건 | 가중치 |
| --- | --- | --- | --- |
| Completion | Assignee 미지정 Task 존재 | 종료 시 | 4 |
| Completion | CSG 외 프로젝트/태스크 존재 | 종료 시 | 3 |
| Completion | CSG에서 canceled:done ≥ 2:8 | 종료 시 | 4 |
| Completion | 프로젝트에 Task 없음 | 종료 시 | 3 |
| Stability | 커밋 외 Task가 Done의 40% 이상 | 종료 시 | 3 |
| Stability | 종료 직전 2일 내 Done 몰림 ≥ 50% | 종료 시 | 3 |
| Stability | Due date 미설정 Task ≥ 30% | 종료 시 | 2 |
| Stability | Sprint Goal 수정 | 수정 시마다 | 3 |
| Process | Sprint 중 삭제된 Task ≥ 30% | 종료 시 계산 | 4 |

※ `스프린트에 프로젝트 없음 && 프로젝트에 task 없음`은 리포트 제외, 점수 미산정

### 2. Sprint 도중 반영 이벤트

#### A. 일정 / 계획 안정성

| 카테고리 | 이벤트 | 조건 | 가중치 |
| --- | --- | --- | --- |
| Stability | Due date 지나서 Done | 1건당 | 2~3 |
| Stability | Due date 연장으로 실제 일정 지연 발생 | `1건 / 2~3건 / 3건 초과` + 영향도 | 미정 |
| Stability | Sprint 기간 변경으로 계획 안정성 하락 | `1건 / 2~3건 / 3건 초과` + 영향도 | 3 |
| Stability | Task 프로젝트 변경으로 재작업/지연 발생 | `1건 / 2~3건 / 3건 초과` + 영향도 | 미정 |

#### B. 상태 흐름

| 카테고리 | 이벤트 | 조건 | 가중치 |
| --- | --- | --- | --- |
| Flow | 리워크 (CSG 이전 상태 회귀) | 1건당 | 3 |
| Flow | TODO → DONE 직행 | 1~3 |  |
| Flow | 특정 상태 장기 방치 (리뷰 병목 포함) | 지속 시 | 4 |
| Flow | Canceled 처리 | 2~4 |  |
| Flow | DONE → IN_PROGRESS / DONE → TODO | `1건 / 2~3건 / 3건 초과` | 4~5 |
| Flow | IN_PROGRESS → TODO / BACKLOG | `1건 / 2~3건 / 3건 초과` | 4 |
| Flow | Sprint 중 Task 추가/삭제 | `1건 / 2~3건 / 3건 초과` | 5 |

#### C. 담당자 / 리소스

| 카테고리 | 이벤트 | 조건 | 가중치 |
| --- | --- | --- | --- |
| Ownership | Assignee 변경으로 인수인계 비용/병목 발생 | `1건 / 2~3건 / 3건 초과` + 영향도 | 미정 |
| Ownership | 특정인 업무 집중도 높음 | 편중 시 | 3 |
| Ownership | 인원 대비 WIP 과다 | 비율 초과 시 | 3 |
| Process | WIP 초과 상태 N일 지속 | 하루당 | 3 |

WIP 기준 초기값:

```text
InProgress / ActiveMembers > 1.2~1.5
```

## 카테고리별 최대 감점 상한

| 카테고리 | Max Penalty |
| --- | --- |
| Flow | 40 |
| Stability | 25 |
| Completion | 25 |
| Ownership | 15 |

> 특정 영역이 크게 무너져도 전체 점수가 과도하게 붕괴하지 않도록 cap을 둔다.

## 최종 점수 계산 구조

미정. 테스트하면서 조정한다.

```text
SprintHealth =
  0.6 × OngoingScore
+ 0.4 × FinalAdjustment
```

## 설계 철학 요약

- 단순 변경보다 변경의 영향에 감점
- 되돌림 최소화
- 병목 제거
- 계획 유지
- 책임 명확화
- 마감 몰아치기 방지
- 예측 가능성 확보
