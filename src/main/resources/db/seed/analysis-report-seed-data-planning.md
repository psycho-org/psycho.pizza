# Analysis Report 기초 데이터 세팅 플랜

## 1) 목표

- `analysis_report` 검증을 위한 기초 데이터(계정/워크스페이스/멤버십) 시나리오를 설계한다.
- 점수 10점 만점 기준으로 최고점/최하점을 양 극단으로 둔 10개 워크스페이스 케이스를 만든다.
- 이후 Sprint 기반 분석(MVP target: `SPRINT`)에서 점수 차이가 재현되도록 데이터 편차를 의도적으로 설계한다.

## 2) 현재 스키마 기준 전제

- 계정 비밀번호는 `accounts.password_hash`에 저장되므로, 원문 비밀번호(`Password1234%^`)를 그대로 넣지 않고 동일 해시 정책으로 저장해야 한다.
  - 애플리케이션 기준 인코더: `BCryptPasswordEncoder`
- 멤버십 역할값은 `OWNER`, `CREW` 2개다.
- 한 워크스페이스 최대 인원은 이번 시드 기준으로 `15명`을 상한으로 둔다.
- 분석 카테고리는 `COMPLETION`, `STABILITY`, `FLOW`, `OWNERSHIP`를 기준으로 케이스를 설계한다.

## 3) Account 30개 구성안

|---|---:|
| Owner | 1 |
| 기획 | 4 |

| 사업운영(비IT) | 3 |

- 오너 제외 직군은 모두 3~5명 범위를 만족한다.
- 비IT 직군(사업운영)을 포함해 IT 편향을 완화한다.

### 3-2. 계정 키 설계

- Owner: `A01`
- 기획: `A02` ~ `A05`
- 설계: `A06` ~ `A09`
- 백엔드: `A10` ~ `A14`
- 프론트: `A15` ~ `A19`
- DevOps/SRE: `A20` ~ `A23`
- QA: `A24` ~ `A27`
- 사업운영: `A28` ~ `A30`

### 3-3. 필드 생성 규칙(권장)

- `email`: `seed+{accountKey}@psycho.local` (예: `seed+a01@psycho.local`)
- `given_name`/`family_name`: 테스트 규칙에 맞춰 일관 생성
- `password_hash`: 모든 계정에 대해 원문 `Password1234%^`를 동일 인코더로 해시한 값 사용

### 3-4. 최신 스키마 메모

- `accounts.display_name` 컬럼은 최신 migration 기준 제거되었고, 이후 `tasks.priority`(`V9`), `memberships.display_name`(`V10`), `sprints.goal`(`V12`), `analysis_metric_count`(`V13`), `reasons`(`V14`)가 추가되었다.
- 따라서 직군 해석은 DB 컬럼이 아니라 `A01~A30` account key 분포표와 `email` 규칙(`seed+aNN@psycho.local`)을 기준으로 본다.

### 4-1. 별도 레퍼런스: 진행 중 워크스페이스 `W00`

- `W00`는 점수 비교용 저/고점 시나리오와 별개로, **현재 스프린트가 진행 중인 상태**를 보기 위한 레퍼런스 워크스페이스다.
- 특징:
  - `2026-03-12` 기준 활성 sprint가 1개 존재한다.
  - 이미 진행 중인 project task와 아직 손대지 않은 `TODO` task가 함께 존재한다.
  - 별도로 `project_task_mapping`이 없는 backlog task가 남아 있어, "프로젝트 미배치 TODO" 상태를 확인할 수 있다.
- 이름 제안: `W00_Current_In_Progress`
- 용도:
  - 완료되지 않은 task가 많더라도 그것이 곧바로 저점 시나리오를 뜻하지 않는 현재 진행형 상태 검증
  - backlog 존재 여부, active sprint 집계, project 미배치 task 집계를 함께 확인하는 레퍼런스

## 6) 실행 순서(기초 데이터 라운드)

1. `accounts` 30건 생성
2. `workspaces` 13건 생성
3. `memberships` 생성 (`Wxx`별 `OWNER/CREW` 매핑)
4. 무결성/분포 검증
   - 이메일 중복 없음
   - 워크스페이스별 인원 15 이하
   - 워크스페이스별 OWNER 정확히 1명

## 7) 후속 라운드(analysis_report 실제 차등 유도)

- 현재 분석 타겟은 `SPRINT`이므로, 점수 차이를 실제로 만들려면 후속으로 아래 데이터가 필요하다.
  - `projects`, `sprints`, `project_task_mapping`, `sprint_project_mapping`
  - `tasks` 상태/마감/지연/담당자 이력
  - 필요 시 `audit_log` 이벤트
- 즉, 이번 문서는 **기초 모수(사람/조직 단위) 설계**까지를 다루고, 다음 단계에서 업무 데이터 편차를 넣어 점수 분포를 완성한다.
